# Reflektion – Git & Agilt (1–2 sidor)

## 1) Egen user story + Acceptance Criteria + INVEST

### US6: Bokstavsordning på bokmärke 

**Story:** Som <användare>  vill jag kunna se enkelt <alla bokmärken i bokstavsordning> så att jag snabbt kan <se strukturerad lista där jag kan hitta enkelt det jag letar efter>.

**Acceptance Criteria (checkboxar):**
- [Alla bokmärken läggs automatiskt till under rätt bokstav ] 
- [Det sparas i localStorage]

**INVEST – kort motivering (2–4 meningar):**
Vilka av I, N, V, E, S, T uppfyller storyn och varför?
- **I**ndependent: Den är inte independent då för att bokmärken ska gå efter bokstqavsordningen ska tidigare funktioner och kod finnas tillgänglig. Även att listan ska kunna renderas först utan problem och etc.
- **N**egotiable:  Den är väldigt förhandlingsbart då man kan diskutera hur man vill utforma den, även stylingen.
- **V**aluable: Den ger värde till användaren såsom att man inte behöver leta efter bokmärken om man har sparat hundratals, även att bokmärken läggs till automatiskt i bokstavsordning.
- **E**stimable: Den kan verkligen uppskattas i antal timmar och tar inte särskilt långt tid.
- **S**mall:  Den kan levereras inom en iteration då den är väldigt small.
- **T**estable: Ja, den kan testas innan och säkerställa att kvaliten och kravet uppfylls.

## 2) Sprintmål + Definition of Done (DoD)
**Sprintmål:** 
Sprintmålen som jag har är att kunna leverera alla **kraven** såsom:
- Merga utan problem
- varje feature landar som 1 commit
- linear history
- göra pull request utan problem
- reviewa pull requesten på klasskamraterna med förbättringar
- implementera förbättringsförslagen i mina commits


**DoD (checklista):**
- [x] Kod kör lokalt utan fel
- [x] PR granskad och godkänd (minst 1 review)
- [x] README uppdaterad
- [x] Issue/kort länkat och stängt vid merge

## 3) Retro: Start / Stop / Continue
- **Start:** (ex. skriva tydligare PR-beskrivningar…)
1. Jag vill börja göra mer reviews. 
2. Inte tycka det är pinsamt och ge feedback. 
3. Börja med projekt i tid. 
4. Skriv **git pull** mycket oftare, inte glömma det
5. Skriv US i varje PR så att det är lätt att följa vilken PR tillhör vilket Trello

- **Stop:** (ex. stora PR över 300 rader…)
1. Inte samla ihop 2 PRs bara för att jag inte orkar vänta på någon ska reviewa min PR. 
2. Inte börja med nya beteenden och nya sätt och göra saker på utan göra saker såsom jag har gjort. Exempelvis så ville jag forka examinationen till mitt repo bara för att jag ville slippa clona ner den. Men det blev grovt-fail.

- **Continue:** (ex. branch-namn `feature/...`)
1. Planera i tid. Sätta mig in i projektet 1 vecka innan och planerar i mitt huvud eller med papper och penna ungefär hur det kommer se ut.
2. Ta hjälp av andra.
3. Fråga en gång för mycket än en gång för lite.
4. Kommentarer på engelska i koden

Minst **en** konkret förbättring du provar nästa sprint och *varför*.
Jag ska inte samla 2 PR i en. Trodde det var inte hela världen men den kan störa så mycket i arbetsgången och göra folk i mitt team förvirrad och även mig själv. Utan följ det man ska göra en enligt agilt arbetsflöde genom att göra 1 PR för sig.
